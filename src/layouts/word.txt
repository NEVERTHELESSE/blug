`<h1><strong>What the fffff…findByIdAndUpdate() mongoose?!</strong></h1><p><a href="https://medium.com/@findingalberta?source=post_page---byline--107219d5f90---------------------------------------" rel="noopener noreferrer" target="_blank" style="color: inherit; background-color: rgb(242, 242, 242);"><img src="https://miro.medium.com/v2/resize:fill:44:44/1*ke5saiLbUBycPBLbyi3AGw.jpeg" alt="Alie Brubaker" height="44" width="44"></a></p><p><span style="color: rgb(36, 36, 36);">·</span></p><p><span style="color: rgb(107, 107, 107);">4 min read</span></p><p><br></p><p>154</p><p><br></p><p>I have very recently started working with Node.js with the intention of learning the MERN stack and to add a little more flexibility to my repertoire of coding skills. In general, it seems that using Express.js and mongoose are extremely common libraries backend developers use to aid them in writing RESTful APIs in Node, so I’ve opted to pick them up.</p><p>Recently, I built a mini project that consisted of a very simple React frontend with an extremely basic user signup form which makes a post request to my Node API, and fetches an index of all users from the database for display. The important thing was to learn the process of building a clean, DRY backend in Node and to write all of the appropriate CRUD actions for the User model.</p><p>Mongoose is intentionally written to make work with a Node API feel like any other backend language/library combination and, having some experience with Ruby on Rails, it did it’s job well. From creating a new Schema (almost exactly like a create_table migration in Rails/SQL) to following a typical MVC structure, mongoose makes Node feel like home.</p><p>While building the controller for my User model I noticed a strange occurrence. When I used the built-in mongoose instance method to update a model&nbsp;<code style="background-color: rgb(242, 242, 242);">.findByIdAndUpdate()</code>&nbsp;, the data Node pulled from my mongoDB server was the original instance of the model, not the newly updated version.</p><p>You might think “So, what’s the big deal?” Well, we generally want to rerender updated resources on the frontend so it is very inconvenient. Furthermore, the other&nbsp;<code style="background-color: rgb(242, 242, 242);">.findById&nbsp;</code>and the&nbsp;<code style="background-color: rgb(242, 242, 242);">.save()</code>&nbsp;methods built into mongoose don’t do the same thing. Both my show route (using&nbsp;<code style="background-color: rgb(242, 242, 242);">.findById()</code>&nbsp;) and my create (using&nbsp;<code style="background-color: rgb(242, 242, 242);">.save()</code>&nbsp;followed by&nbsp;<code style="background-color: rgb(242, 242, 242);">.then()</code>&nbsp;) pulled the version that existed currently in the database.</p><p><img src="https://miro.medium.com/v2/resize:fit:582/1*K5UXvV826gYy474I3I82Yw.png" alt="a code block showing a node.js controller action" height="249" width="582"></p><p>the create action for the user controller</p><p><img src="https://miro.medium.com/v2/resize:fit:583/1*6BwojcgFG3k5T-7Wn8EHNw.png" alt="A terminal readout for a server showing a console log" height="173" width="583"></p><p>the console.log() of the data that .save() returned</p><p>So .findByIdAndUpdate() was updating the database but it was returning the old data that we just replaced out in response to the promise.</p><p><img src="https://miro.medium.com/v2/resize:fit:583/1*_2XiWWVW89z__-hw72GLbg.png" height="232" width="583"></p><p>the update action for the user controller</p><p><img src="https://miro.medium.com/v2/resize:fit:700/1*WjOax3zdczsQ5zy1WK6Ujw.png" height="522" width="700"></p><p>a Postman entry to patch the user</p><p><img src="https://miro.medium.com/v2/resize:fit:580/1*evC10V0iN9O8jUTb4EJmbw.png" height="186" width="580"></p><p>the console.log() for the stale data that .findByIdAndUpdate() returned</p><p><img src="https://miro.medium.com/v2/resize:fit:633/1*WuONw1tp_U06_VB2Dk29TQ.png" height="176" width="633"></p><p>the actual data in mongoDB Atlas</p><p>As you can see, the data WAS in fact updated by the method it just returned the old data. I tried a dozen different ways of writing the method including just finding the instance by id, saving it and pulling the data asynchronously like I did with the create but for whatever reason even that didn’t work.</p><p>The solution was extremely simple. .findByIdAndUpdate() can take four arguments. The first is the id of the record in the database. The second is the body pulled out of the request object. The third, which was not immediately apparent in the docs, is a configuration object with a number of attributes. The final parameter is a function fired asynchronously that takes the return data or an error instance for parameters.</p><p>The configuration object simply needs to read&nbsp;<code style="background-color: rgb(242, 242, 242);">{new: true}</code>&nbsp;to automatically return the new entry. That’s it. All I could do is laugh.</p><p><img src="https://miro.medium.com/v2/resize:fit:598/1*vfISDbPKnknxMIxcDOxzeA.png" height="236" width="598"></p><p>the working update action for the user controller</p><p><img src="https://miro.medium.com/v2/resize:fit:700/1*2uvBsQfGb7ULpeH8uolbPg.png" height="518" width="700"></p><p>a Postman entry to patch the user</p><p><img src="https://miro.medium.com/v2/resize:fit:540/1*6-VaPWMC80HemtMRLLpnnQ.png" height="193" width="540"></p><p>the console.log() for the correct data that .findByIdAndUpdate() returned</p><p>It really is that simple. Call&nbsp;<code style="background-color: rgb(242, 242, 242);">.findByIdAndUpdate( id, req.body,&nbsp;<strong>{new: true}</strong>, (err, data) =&gt; {…})</code>&nbsp;and you’re aces!</p><p>***************************************************</p><p>I’m looking for work as a software engineer.
